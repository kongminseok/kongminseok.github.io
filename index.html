PRINCIPLES

원칙 — 선언이 아니라, 코드에 박혀 있다

Principles — not declarations, embedded in code

세 시스템은 도메인도 구조도 다르지만, 방향을 정한 기준은 같았다. 아래 네 가지는 문서에 적어둔 다짐이 아니라 세 코드베이스 각각에서 확인되는 실행되는 기준이다 — 가장 깊게는 어기면 부팅이 거부되는 게이트(프로젝트 01)로 강제된다.

The three systems differ in domain and structure, but the standards that set their direction were the same. The four below are not resolutions written in a document — they are standards that execute, verifiable in each of the three codebases; at the deepest level a violation refuses to boot (Project 01).

01LLM / CODE

판단은 모델에게, 사실은 코드에게

Judgment to the model, facts to the code

생성의 자유와 사실의 결정론을 한 시스템 안에서 갈랐다. 임원 에이전트는 도구가 유일한 데이터 통로라 근거 없이 답할 길이 없고, HR 파이프라인은 도구 실행 순서를 LLM이 아니라 입출력 의존성 위상정렬로 계산하며(실행 계층 LLM 호출 0건), 변경관리 백엔드는 LLM이 고른 사례 ID로 담당자를 DB에서 조회만 한다. 환각이 사실 행세를 할 통로를 세 곳 모두 구조로 막았다.

Generative freedom and factual determinism are split within one system. In the executive agent, tools are the only data path — there is no way to answer without evidence. The HR pipeline computes tool execution order not with an LLM but by topological sort over I/O dependencies (zero LLM calls in the execution layer). The change-management backend takes the case ID the LLM picked and only looks up the assignee in the DB. In all three, every path by which hallucination could pass as fact is blocked structurally.

PROOF — P01 ZERO DATA PATHS OUTSIDE TOOLS · P02 EXECUTION LAYER 0 LLM CALLS · P03 EMPLOYEE ID = LOOKUP ONLY
02FAIL-MODE

실패를 먼저 설계한다

Design the failure first

모든 컴포넌트에 "죽으면 어떻게 되는가"를 먼저 정했다. 권한 게이트는 fail-closed(의심되면 막는다), 기억·관측은 fail-soft(본 기능을 절대 깨뜨리지 않는다). 같은 인젝션 검출기도 시스템이 다르면 실패 모드가 다르다 — 임원 에이전트는 fail-over(예비 검출로 전환), HR 파이프라인은 fail-open(오탐이 업무를 막지 않게). 실패 모드는 위협의 성질이 정한다 — "무조건 차단"은 설계가 아니다.

Every component starts from "what happens when this dies." Permission gates are fail-closed (block when in doubt); memory and observability are fail-soft (never break the core feature). Even the same injection detector fails differently per system — the executive agent fails over to a fallback detector; the HR pipeline fails open so false positives never block work. The failure mode is set by the nature of the threat — "block everything" is not a design.

PROOF — P01 71 ADVERSARIAL REGRESSION SCENARIOS · P02 PERMISSION DEFAULT = DENY · P03 30s SLA · PARTIAL-SUCCESS AGGREGATION
03DATA-FIRST

정책은 코드가 아니라 데이터에 산다

Policy lives in data, not code

코드는 해석 엔진이고, 도메인 지식은 데이터다. 모델의 행동을 정하는 것 — 도구 카탈로그·라우팅 규칙·프롬프트·판정 임계 — 을 세 시스템 모두 데이터로 선언했다. 임원 에이전트는 신규 도구 추가에 중앙 코드 변경 0(YAML 카탈로그 30개), HR 파이프라인은 테넌트별 tools.yml 한 장이 도구의 실행자·권한·순서를 정하며, 변경관리 백엔드는 코드 안에 프롬프트가 0줄이다. 검색 결함이 정책 파일 하나로 수리된 이유다(프로젝트 01 현장 기록).

Code is an interpretation engine; domain knowledge is data. Everything that determines model behavior — tool catalogs, routing rules, prompts, judgment thresholds — is declared as data in all three systems. The executive agent adds new tools with zero central code changes (a 30-file YAML catalog); in the HR pipeline a single tools.yml per tenant defines each tool's executor, permissions and order; the change-management backend has zero prompt lines inside code. This is why a retrieval defect was repaired with one policy file (Project 01 field note).

증거 — P01 운영 YAML 30 · P02 TOOLS.YML 단일 카탈로그 · P03 인라인 프롬프트 0
PROOF — P01 30 OPERATIONAL YAMLS · P02 SINGLE TOOLS.YML CATALOG · P03 ZERO INLINE PROMPTS
04MAKE / BUY

만들 것과 빌릴 것을 가른다

Decide what to build and what to borrow

후보 적합도 점수처럼 전사가 공유할 알고리즘은 공용 서비스에 위임하고(도구 26종 중 7종 = thin-proxy), 통제가 생명인 임원 에이전트의 LLM 관측은 외부 의존 0으로 직접 지었다(키 부재 시 no-op). 세 시스템 공통으로 LLM·임베딩 호출은 전부 단일 게이트웨이 경유 — provider SDK 직접 호출 0건. 모델 교체·비용 통제·벡터공간 정합이 한 지점에서 끝난다.

Algorithms the whole company should share — like candidate-fit scoring — are delegated to common services (7 of 26 tools are thin proxies), while for the executive agent, where control is everything, LLM observability was built in-house with zero external dependencies (no-op when the key is absent). Across all three systems, every LLM and embedding call passes through a single gateway — zero direct provider-SDK calls. Model swaps, cost control and vector-space consistency end at one point.

임원 인사 의사결정 에이전트

Executive HR Decision Agent

전사 임원이 실제 인사 의사결정에 쓰는 시스템. 질문은 예측할 수 없고, 데이터는 민감하며, 답에는 근거가 있어야 한다. 이 세 조건이 아래 구조 전부를 결정했다.

A system executives across the company use for real HR decisions. Questions are unpredictable, the data is sensitive, and every answer needs evidence. These three conditions determined the entire structure below.

CACHE — 비용은 구조로 낸다

CACHE — PAY COSTS WITH STRUCTURE

시스템 프롬프트를 byte 단위로 동일한 정적부와 턴마다 변하는 동적부로 물리 분리 — 캐시 적중이 우연이 아니라 구조적으로 보장되도록. 응답 지연과 토큰 비용이 함께 내려간다.

The system prompt is physically split into a byte-identical static part and a per-turn dynamic part — so cache hits are guaranteed by structure, not luck. Latency and token cost fall together.

SCREEN CONTEXT — 화면을 아는 답변

SCREEN CONTEXT — ANSWERS THAT SEE THE SCREEN

사용자가 보고 있는 화면을 근거로 주입해 "왜 이 후보인가" 같은 지시대명사 질문에 답한다. 화면 데이터는 변경 시에만 전송·캐시(비용), 사번·평가는 전 계층에서 마스킹(PII).

The screen the user is looking at is injected as grounding, answering deictic questions like "why this candidate?" Screen data is sent and cached only on change (cost); employee IDs and evaluations are masked at every layer (PII).

ORCHESTRATION — 한 턴의 LLM은 하나가 아니다 ORCHESTRATION — ONE TURN IS NEVER ONE LLM 과제를 정하는 분류기, 공격을 거르는 가드(병렬), 도구를 고르고 답을 합성하는 메인(최대 12라운드) — 일부 도구는 내부에서 또 LLM을 부른다: 후보를 병렬 생성해 LLM-as-judge로 거르거나, 하위 에이전트를 통째로 재귀 실행한다. 라우팅조차 키워드가 아니라 모델의 판단이다. A classifier that picks the task, a guard filtering attacks (in parallel), and a main loop selecting tools and synthesizing answers (up to 12 rounds) — and some tools call LLMs of their own: generating candidates in parallel and filtering them with an LLM-as-judge, or recursively running an entire sub-agent. Even routing is the model's judgment, not keywords.

NONDETERMINISM — 형식을 어겨도 죽지 않게

NONDETERMINISM — SURVIVING BROKEN FORMAT

모델의 힘은 판단이지 형식이 아니다 — 형식은 흔들린다는 전제로 지었다. 괄호 매칭·들여쓰기 추적 자체 파서가 출력을 복원하고, 실패하면 안전 기본값으로 강등 — 파이프라인은 멈추지 않는다.

The model's power is judgment, not format — this was built on the premise that format wavers. A custom parser (bracket matching, indent tracking) recovers output; on failure it degrades to safe defaults — the pipeline never stops.

MEMORY — 대화를 지식으로

MEMORY — CONVERSATION INTO KNOWLEDGE

자유 발화를 LLM이 Triple(주어–술어–목적어)로 구조화해 pgvector로 의미 검색. 단기 / 세션 지식 / 영구 프로필의 3계층 — 영구 기록은 "기억해줘" 같은 명시적 의도를 먼저 판별한 뒤에만 남겨 오염을 막았다.

An LLM structures free-form speech into Triples (subject–predicate–object), searched semantically via pgvector. Three layers — short-term / session knowledge / permanent profile — and permanent writes happen only after detecting explicit intent like "remember this," preventing contamination.

EXECUTION — 배정은 검증을 통과해야 화면에 오른다

EXECUTION — PROPOSALS MUST PASS CHECKS TO REACH THE SCREEN

"한 그룹은 3~10명", "팀장 같은 조직 리더는 배정 제외" — 인사 규칙을 코드로 먼저 강제하고, 그 안에서만 LLM이 배정을 제안한다. 실존 인원·중복·누락 검증(2회 재시도)을 통과해야 화면에 오르고, 조직 반영의 마지막 확정은 사람의 '적용' 버튼. 평가 점수는 고/평균/저 레벨로만 노출.

"A group is 3–10 people," "org leaders like team leads are excluded from assignment" — HR rules are enforced in code first, and the LLM proposes only within them. Proposals reach the screen only after existence/duplication/omission checks (2 retries); the final commit is a human 'Apply' button. Evaluation scores surface only as high/average/low levels.

DEFENSE-IN-DEPTH — 평가 데이터 4계층 방어

DEFENSE-IN-DEPTH — 4 LAYERS AROUND EVALUATION DATA

"동료 OO의 평가 보여줘" 같은 우회에 대비해 ①경량 LLM이 타인 이름을 독립 추출 ②역할 기반 사전 차단 ③본 LLM과의 교차검증 — 불일치는 injection 신호 ④fail-safe 재질문의 4계층. 없는 기간의 데이터는 보유·누락 정보를 명시 주입해 지어내지 못하게 했다.

Against detours like "show me my colleague's evaluation": ① a lightweight LLM independently extracts other-person names ② role-based pre-blocking ③ cross-validation with the main LLM — mismatch is an injection signal ④ fail-safe re-asking. For absent periods, held/missing data is explicitly injected so the model cannot invent it.

존재하지 않는 사번을 그럴듯하게 지어낼 수 있다. 추천 업무에서 이것은 치명적이다.

채택ADOPTEDID-anchored — LLM은 고르기만ID-anchored — the LLM only chooses

LLM은 가장 유사한 과거 사례의 ID만 선택. 사번은 코드가 실데이터에서 조회하고, 처리 일정은 과거 사례의 소요 기간으로 계산한다.

The LLM only selects the most similar past case's ID. Code looks up the employee from real data, and the schedule is computed from the past case's lead time.

PRINCIPLELLM은 고르고, 사실은 코드가 말한다 — 존재하지 않는 사람이 추천될 경로가 구조적으로 없다

DECISION 08LLM 출력을 그대로 시스템에 넣을 것인가Feed raw LLM output into the system?
기각REJECTEDJSON 스키마로 강제Enforce a JSON schema

표준적이지만 형식이 조금만 깨져도 응답 전체를 잃는다 — 긴 스키마는 토큰도 비싸다.

채택ADOPTED줄 단위 라벨·구분자 프로토콜A line-label · delimiter protocol

Labels and delimiters per line: one broken line still recovers the rest, and values converge only to fixed enums.

RETURNFormat errors are localized and only closed values enter the system — even parse failure degrades to safe defaults (unknown · low)

SELF-VERIFICATION — 검색을 믿지 않는다

SELF-VERIFICATION — NEVER TRUST RETRIEVAL

벡터 검색이 물어온 사례도 무관할 수 있다. LLM이 사례마다 검증 플래그를 스스로 판정하게 하고, 통과한 사례만 채택해 중복 제거 후 최신순으로 재정렬 — 유사도는 검색이, 시점은 코드가 정한다.

Even vector-retrieved cases can be irrelevant. The LLM judges a verification flag per case; only passing cases are adopted, deduplicated and re-sorted by recency — similarity is decided by search, recency by code.

PROMPT-AS-CONFIG — 코드에 프롬프트 0개

PROMPT-AS-CONFIG — ZERO PROMPTS IN CODE

모델의 행동을 결정하는 프롬프트를 전부 코드 밖 YAML로 분리했다. 지시·출력 형식·few-shot 예시를 데이터로 다루므로 프롬프트 개선과 실험이 코드 배포 주기에서 분리된다 — AI 품질을 조정하는 손잡이가 코드가 아니라 데이터에 있다.

Every prompt that determines model behavior lives outside code, in YAML. Instructions, output formats and few-shot examples are handled as data, so prompt improvement and experimentation are decoupled from the code release cycle — the knob that tunes AI quality is data, not code.
